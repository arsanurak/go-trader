และ ADX

Fold 5 ของ squeeze_momentum ล้มเหลวด้วย Sharpe −2.64 — ไม่ใช่เพราะ strategy เสีย แต่เพราะ market อยู่ใน regime ผิดประเภท เรียนรู้ว่า ADX คืออะไร go-trader ใช้มันอย่างไร และ allowed_regimes ป้องกันปัญหาแบบนี้ได้อย่างไร

  1. อ่านค่า ADX, DI+, DI− แล้วบอกได้ว่า market อยู่ใน regime ใด
  2. เข้าใจว่า allowed_regimes gate ทำงานอย่างไรใน go-trader config
  3. อธิบายได้ว่าทำไม fold 5 จึงล้มเหลว และ regime gate จะช่วยได้อย่างไร

กลับไปที่ Fold 5

ใน Lesson 04 เราพบว่า BTC/USDT 1h ของ squeeze_momentum มีปัญหาใน fold สุดท้าย:

Fold ช่วงเวลา Sharpe Return Trades
1–4Jun 2023 – May 2025+0.89 ถึง +3.36+3.6% ถึง +17.3%10–17
5Oct 2025 – Jan 2026−2.64−10.1%2

Squeeze momentum เป็น breakout strategy — มันรอ price consolidate แล้ว break out ไปทิศทางหนึ่ง มันทำงานดีเมื่อ market กำลัง trending — มี momentum ชัดเจน แต่เมื่อ market ไม่มี trend (choppy, sideways) breakout สัญญาณจะ false สูงมาก ราคาออกมาแล้วกลับมาทันที — เรียกว่า whipsaw

สมมติฐาน

Fold 5 เป็นช่วงที่ BTC ไม่มี trend ชัดเจน market consolidate อยู่นาน Squeeze momentum ได้รับสัญญาณ breakout แต่ breakout ทุกครั้งล้มเหลว N=2 บอกว่า trades น้อยมาก — ถ้า regime gate เปิดใช้ อาจ skip เข้าทั้งหมด

คำถามคือ: เราจะ วัด ได้อย่างไรว่า market กำลัง trend หรือไม่? คำตอบคือ ADX

ADX คืออะไร

ADX (Average Directional Index) วัด ความแรงของ trend — ไม่ใช่ทิศทาง ค่า ADX สูง = trend แรง ไม่ว่าจะขึ้นหรือลง ค่า ADX ต่ำ = market choppy ไม่มีทิศทาง

ADX คำนวณจากสองค่าเสริมคือ DI+ (Directional Indicator ขาขึ้น) และ DI− (ขาลง) ทั้งสามตัวนี้คือสิ่งที่ go-trader บันทึกทุก cycle ใน shared_tools/regime.py

0 20 40 100
Ranging
ADX < 20 อาจ trend
ADX 20–40 Trend แรง
ADX > 40

go-trader ใช้ default adx_threshold = 20.0 (config field regime.adx_threshold) เป็นเส้นแบ่ง: ต่ำกว่า 20 = ranging, สูงกว่า 20 = ดู DI+ กับ DI− เพื่อหาทิศทาง

กฎ: ตรวจ ADX ก่อนเสมอ — ถ้า ADX ต่ำกว่า threshold ให้ตอบ ranging ทันที ไม่สนใจค่า DI+ DI− เลย (แม้ DI+ จะสูงกว่า DI− มาก)

go-trader codebase

Logic นี้อยู่ใน shared_tools/regime.py ฟังก์ชัน compute_regime() บรรทัด 271–282: loop ผ่านทุก bar, เช็ค adx_val < adx_threshold ก่อน แล้วค่อยเปรียบเทียบ plus_di[i] > minus_di[i]

ค่า adx_threshold default คือ 20.0 (บรรทัด 77 ใน scheduler/config.go) ปรับได้ใน top-level config: regime.adx_threshold

เปิดใช้ Regime Detection ใน Config

Regime detection ปิดอยู่ by default ต้องเปิดใช้ที่ระดับ top-level config และตั้ง allowed_regimes ที่ strategy เพื่อให้ gate ทำงาน:

// scheduler/config.json — เพิ่มสองส่วนนี้ { "regime": { "enabled": true, // เปิดระบบ ADX ทั้งหมด "period": 14, // Wilder's ADX period (default 14) "adx_threshold": 20.0 // ต่ำกว่านี้ = ranging }, "strategies": [{ "id": "hl-squeeze_momentum", "platform": "hyperliquid", ... (config อื่นๆ) "allowed_regimes": [ "trending_up", "trending_down" ] // เข้า trade เฉพาะเมื่อ market trending เท่านั้น }] }

ถ้าตั้ง allowed_regimes ไว้แต่ลืมเปิด regime.enabled, go-trader จะแสดง warning ว่า gate เป็น no-op — ดู scheduler startup log

go-trader codebase

Regime label ถูกส่งเข้า Go scheduler ผ่าน JSON stdout ของ check_*.py Go อ่านใน scheduler/regime_atr.go และเปรียบเทียบกับ AllowedRegimes []string (config field allowed_regimes ใน scheduler/config.go บรรทัด 362)

กฎสำคัญ: allowed_regimes gate entries เท่านั้น — ถ้า position เปิดอยู่แล้ว close จะยังทำงานตามปกติแม้ regime เปลี่ยน

ทำไม Regime Gate ช่วย Fold 5 ได้

Fold 5 (Oct 2025 – Jan 2026) มี N=2 trades เท่านั้น — หมายความว่า ถ้า allowed_regimes: ["trending_up", "trending_down"] เปิดใช้อยู่, strategy อาจ skip entries ทั้งหมดในช่วงนั้น เพราะ ADX ต่ำกว่า 20 ตลอด

ผลลัพธ์คือ: แทนที่จะขาดทุน −10.1% จาก 2 trades ที่ whipsaw, strategy จะไม่มี trades เลยในช่วงนั้น — equity ไม่เปลี่ยนแปลง

Insight สำคัญ

"ไม่เข้า trade" อาจเป็น outcome ที่ดีที่สุดในบาง regime Regime detection ไม่ได้หา edge ใหม่ — มันป้องกันไม่ให้ strategy ทำงานในสภาวะที่มันไม่ได้ถูกออกแบบมาเพื่อ

แต่มีข้อแลกเปลี่ยน: การ filter ออก regime บางประเภทลด N trades รวม ถ้า N ลดลงมากเกินไป ตัวเลข Sharpe จะไม่ reliable (กลับไป lesson 03!) ต้องทดสอบว่า filtered backtest ยังมี N เพียงพอหรือไม่
